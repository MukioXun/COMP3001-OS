fork() 系统调用 ↑

  • 调用一次，返回两次：父进程得子 PID，子进程得 0
  • 子进程复制父进程的变量（写时复制），之后互不影响
  • 子进程从 fork 返回处开始执行；子进程 PID 不一定大于父进程
  • 同步父子顺序：在第二个 fork 前加 wait(NULL)

计算套路：fork 进程数与输出 ↑

数进程：顺序执行的 n 个 fork → 最多 2ⁿ 个进程；带条件分支看进程图

数输出：每个 printf 在每个经过它的进程中各执行一次

例（作业一第 11 题）：两次 fork，每次只在父进程继续 → 共 3 个进程，x 输出 0、1、3（子进程退出 exit(0) 不继续）

例（作业一第 12 题）：L0 fork (L1 fork (L2 Bye) Bye) Bye → 4 个进程，输出含 1×L0、1×L1、1×L2、3×Bye

数 Bye 的次数 = 最终进程总数；不要漏掉 fork 返回后父子都执行的代码

进程三态 ↑

  • 就绪→运行：被调度选中
  • 运行→就绪：时间片用完 / 高优先级抢占
  • 运行→阻塞：等待 I/O 或事件
  • 阻塞→就绪：等待事件发生（不能直接到运行）
  • 僵尸态：子进程已退但父未 wait，PCB 未回收（非正常状态）

进程 vs 线程 ↑

  • 进程是资源分配基本单位，线程是 CPU 调度基本单位
  • 同一进程的线程共享地址空间、文件、全局变量，独立栈和寄存器
  • 线程切换比进程切换快（不切地址空间）
  • 不管有无线程，进程都是资源拥有的独立单位

PCB 进程控制块 ↑

  • 进程存在的唯一标志，保存在内存的系统区（不在用户空间，用户进程无法直接访问）
  • 存：PID、状态、寄存器副本、调度信息、资源清单、页表指针等
  • 不存代码和数据本身（只存指向它们的地址或描述符）
  • 进程切换 = 保存当前 PCB + 恢复目标 PCB
展开：进程切换的最核心实现

上下文切换（context switch）的硬件级步骤：

  1. 保存现场到当前 PCB：
    • 通用寄存器（rax/rbx/...）
    • 程序计数器 PC（要执行的下一条指令）
    • 栈指针 SP（用户栈和内核栈位置）
    • 程序状态字 PSW（标志位、CPU 模式）
  2. 切换内存映射：把页表基址寄存器（CR3）改为目标进程的页表根 → 整个地址空间被替换
  3. 刷新 TLB（或仅刷无 PCID 标记的项）—— 因为虚拟地址含义变了
  4. 恢复现场：从目标 PCB 加载寄存器、PC、SP、PSW
  5. 从目标 PC 处继续执行 —— 完成切换

开销主要来自：①TLB 刷新导致后续大量 cache miss；②内核态进入退出；③页表切换。这就是线程切换比进程切换快的根本原因 —— 线程不切页表也不刷 TLB。

线程切换只需保存通用寄存器+PC+SP，不切 CR3；进程切换则全套都要

线程模型 ↑

  • 多对一（用户级）：调度快，但一线程阻塞整进程阻塞，不能并行
  • 一对一（内核级）：可并行，切换开销大
  • 多对多：用户级线程映射到 ≤ 数量的内核线程，折中
展开：用户级 vs 内核级线程详细对比
维度用户级线程 ULT内核级线程 KLT
调度者用户态线程库操作系统内核
切换开销小（不进内核）大（陷入内核）
多核并行❌ 不能✅ 能
阻塞影响整进程一起阻塞仅该线程阻塞
对内核透明✓ 内核不知道✗ 内核管理
典型实现早期 Green ThreadsPOSIX pthread (Linux NPTL)

三种映射模型：

  • M:1（多对一）：M 个用户线程 → 1 个内核线程。所有 ULT 共用一个内核调度实体；任一阻塞拖累全体
  • 1:1（一对一）：每 ULT 配一个 KLT。现代主流（Linux/Windows）；可真并行
  • M:N（多对多）：M 个 ULT 复用 N 个 KLT（N < M）。理论灵活但实现复杂，已较少使用

"线程是 CPU 调度的基本单位"这句话默认指内核级线程；用户级线程实际调度单位仍是其所在进程

进程间通信 ↑

  • 管道 pipe：半双工；无名管道仅亲缘进程；有名管道可任意
  • 共享内存：最快，但需同步
  • 消息队列、信号量、socket（跨主机）

四类异常 ↑

中断 Interrupt外部硬件异步返回下一条I/O 完成、时钟
陷阱 Trap故意指令同步返回下一条系统调用、断点

缺页是 故障，处理完后重新执行原指令（这就是 TLB 题里要算两次访问的原因）

异常处理机制 ↑

  1. 事件发生 → CPU 检测到异常号
  2. 查异常表（异常号 → 处理程序入口）
  3. 保存上下文（PC、寄存器、PSW）压栈
  4. 切换到内核态，执行处理程序
  5. 处理完成 → 恢复上下文 → 返回用户态

异常表与中断向量表是同一类结构，按异常号索引。

系统调用 ↑

  • 本质：一种陷阱，用户主动进入内核的唯一合法方式
  • 流程：用户调库函数 → 库函数把参数和系统调用号放寄存器 → 执行 syscall/int 指令 → 进内核执行 → 返回结果
  • fork、read、write、open、wait 都是系统调用

信号 Signal ↑

定义：内核或进程发送给目标进程的小整数消息，软件层面的异常控制流。是 IPC 的一种机制。

常见信号：

  • SIGINT(2) Ctrl+C 中断
  • SIGKILL(9) 强杀，不可捕获/忽略/阻塞
  • SIGSEGV(11) 段错误
  • SIGALRM(14) 定时器到期
  • SIGCHLD(17) 子进程停止/终止
  • SIGSTOP 暂停，不可捕获/忽略/阻塞

三种处理方式：忽略 / 终止（默认）/ 捕获（自定义 handler）

发送方式：kill(pid,sig)系统调用 / kill -9 pid命令 / 键盘快捷键 / alarm(n)定时

信号接收机制：两个位向量 ↑

内核为每个进程维护两个位向量：

  • pending（待处理）：信号到达置 1，处理后清零
  • blocked（阻塞）：进程主动屏蔽的信号集合
实际处理的信号 = pending & ~blocked

检查时机：进程从内核态切回用户态时

阻塞方式：

  • 隐式：执行某信号 handler 时，内核自动阻塞同类信号
  • 显式：用 sigprocmask(SIG_BLOCK/UNBLOCK/SETMASK, ...)

信号不排队！同类信号多次到达时，pending 位向量只记录"有一个待处理"，多余信号被丢弃

安全的信号处理（async-signal-safe） ↑

handler 与主程序并发运行，必须遵守五条准则：

  • G1 简单处理：handler 尽量只设置标志位
  • G2 仅用 async-signal-safe 函数：write() 安全；printf/malloc/exit 不安全
  • G3 保存/恢复 errno：handler 开始保存、结束恢复
  • G4 访问共享数据前阻塞所有信号：防竞争条件
  • G5 全局变量加 volatile：防编译器缓存到寄存器；标志位用 sig_atomic_t 类型

在 handler 里调用 printf/malloc 是经典 bug：可能死锁或数据损坏（因为这些函数有内部锁/共享缓冲区）

可移植性：用 sigaction() 替代 signal()（后者行为在不同系统不一致），配合 SA_RESTART 标志和 sigemptyset() 初始化

中断 vs 轮询 ↑

  • 轮询：CPU 主动反复查状态，浪费 CPU；适合高频事件
  • 中断：设备就绪时主动通知 CPU，CPU 可并行其他工作；适合低频事件

信号量本质 ↑

信号量 S = 带等待队列的整数。

  • P(S)：S--；若 S<0，进程阻塞入队
  • V(S)：S++；若 S≤0，唤醒一个等待进程
  • 三种用途：互斥（初值 1）、同步（初值 0）、资源池（初值 n）

信号量值的物理含义 ↑

  • S > 0：表示剩余可用资源数
  • S = 0：资源恰好用完，无等待
  • S < 0：|S| 表示当前等待该资源的进程数

例：初值 5，当前值 -2 → 资源已用完，2 个进程在等

例：初值 3，当前 -1 → 1 个进程在等

例：3 进程共享、最多 2 个并发 → 初值 2，范围 [2, -1]

互斥 vs 同步判别 ↑

  • 互斥：P 和 V 在同一进程中，包住临界区；初值通常 1
  • 同步：P 和 V 在不同进程中（交叉）；初值通常 0
  • 判别：写完后看每个信号量，如果 P/V 总成对出现在同一进程 → 互斥用；否则 → 同步用

计算套路：设计 PV 同步 ↑

四步法：

  1. 识别"事件 / 等待关系" → 每条 → 一个初值 0 的同步信号量
  2. 识别"互斥访问的共享变量" → mutex = 1
  3. 识别"有限资源" → 信号量初值 = 资源数
  4. 写代码：临界区用 P(mutex)…V(mutex) 包；同步事件 V 在前驱末尾、P 在后继开头

永远不要"持有 mutex 时去 P 一个可能阻塞的信号量" → 死锁

计算例题：前驱图同步（作业一第 9 题） ↑

C 必须在 A、B 之后；E 必须在 C、D 之后。

信号量：sem_A=sem_B=sem_C=sem_D=0

A: { A(); V(sem_A); }
B: { B(); V(sem_B); }
C: { P(sem_A); P(sem_B); C(); V(sem_C); }
D: { D(); V(sem_D); }
E: { P(sem_C); P(sem_D); E(); }

规则：每条依赖边 → 一个信号量；后继操作开头连续 P 所有前驱信号量

复杂同步题结构 ↑

多状态、多队列题（如录像厅、理发店）的统一模板：

// 共享状态变量（多个）
mutex = 1              // 保护所有状态变量
条件信号量1 = 0        // 等待事件1
条件信号量2 = 0        // 等待事件2

进入逻辑:
  P(mutex)
    if (能立即进) { 改状态; V(mutex); }
    else {
      登记等待数;
      V(mutex);          ← 必须先释放 mutex
      P(条件信号量);     ← 再去等
    }

退出逻辑:
  P(mutex)
    改状态;
    if (我是最后一个) {
      批量 V 唤醒等待者;
    }
  V(mutex)

计算例题：录像厅同步（作业一第 10 题） ↑

关键设计：

  • current（当前类型）+ count（人数）+ waiting_X（各类等待数）—— 共享变量，全部用 mutex 保护
  • 三个等待信号量 theater_A/B/C = 0
  • 进入时若类型匹配立即进；否则登记+释放 mutex+P 等待
  • 退出时若是最后一个 → 切换 current → 批量 V 唤醒新类型的等待者

最后一个观众负责"切场"。批量唤醒数 = waiting 数，唤醒后清零计数

线程内存模型 ↑

同进程的所有线程共享地址空间，但变量是否真共享要看声明位置：

变量类别实例数是否共享位置
全局变量1 个所有线程共享数据段
局部 auto每线程 1 个各自独立各线程的栈
局部 static1 个所有线程共享数据段

局部 auto 在线程私有栈上，但把栈地址传给其他线程就被共享了（危险！） — 这是典型的"栈逃逸"bug

竞争条件 Race Condition ↑

典型例子：两线程各做 cnt++ 1000 次，期望 2000，实际可能少很多。

cnt++ 不是原子的，对应三条汇编：

L (Load):   cnt → 寄存器
U (Update): 寄存器 + 1
S (Store):  寄存器 → cnt

两线程的 L/U/S 交错执行时更新丢失：

线程A: L(读cnt=5)        U(=6)        S(写6)
线程B:           L(也读到5)  U(=6)         S(写6)
结果：cnt = 6（应该是 7）

消除方法：原子操作 / 锁 / 信号量

临界区四原则 ↑

  1. 空闲让进：临界区空时，允许一进程立即进入
  2. 忙则等待：临界区有进程时，其他必须等
  3. 有限等待：等待时间有上限（防饥饿）
  4. 让权等待：不能进入时应放弃 CPU，不能忙等

违反"让权等待"的算法（如自旋锁）只适合短临界区。

读者-写者问题（必背模板） ↑

第一类（读者优先）：只要有读者在读，新读者直接进 → 写者可能饥饿

int readcnt = 0;
semaphore mutex = 1;   // 保护 readcnt
semaphore w = 1;       // 写者互斥锁

Reader:                          Writer:
P(mutex);                        P(w);
  readcnt++;                     写;
  if (readcnt == 1) P(w); // 第一个读者抢锁
V(mutex);                        V(w);
读;
P(mutex);
  readcnt--;
  if (readcnt == 0) V(w); // 最后一个读者释放
V(mutex);

第二类（写者优先）：写者在等时不允许新读者进入 → 加一个 readMutex，写者持有它阻挡新读者

读优先 → 写者饥饿；写优先 → 读者饥饿；公平方案需用队列

哲学家就餐问题（三种解法） ↑

5 哲学家围坐，每人需左右两根筷子才能吃。错误方案：所有人同时拿左筷 → 全等右筷 → 死锁（四条件全满足）

解法 1：限制人数

semaphore sm = 4;   // 最多 4 人同时尝试

Philosopher i:
P(sm);
  P(chopstick[i]);
  P(chopstick[(i+1)%5]);
  吃;
  V(chopstick[i]); V(chopstick[(i+1)%5]);
V(sm);
// 至少有一人能吃饭，破坏"循环等待"

解法 2：原子取双—— 用 mutex 包住两次 P，只有两筷都可用才拿

解法 3：奇偶序号—— 奇数号先左后右，偶数号先右后左，打破循环等待

线程安全 ↑

四类非线程安全函数：

  1. 不保护共享变量的函数（如 badcnt）
  2. 调用间保持状态的函数（如 rand() 用静态种子）
  3. 返回指向静态变量指针的函数（如 ctime()、gethostbyname()）
  4. 调用非线程安全函数的函数

可重入函数：不引用任何共享数据；是线程安全的子集

解决方案：用 _r 后缀的可重入版本（如 ctime_r）；或自己加锁

  • 关中断：单 CPU 简单方案；多 CPU 失效

三个核心公式 ↑

周转时间 = 完成时间 − 到达时间
带权周转 = 周转时间 / 服务时间

响应比 = (等待时间 + 服务时间) / 服务时间

调度算法对比 ↑

算法抢占饥饿特点
FCFS否否公平，短作业等久
SJF/SRTN可抢占会平均周转最短

展开：每个算法的设计动机

FCFS（先来先服务） — 最朴素直觉，按到达排队
缺点：一个长作业卡住所有短作业（"护航效应"）
适用：批处理、追求公平的系统

SJF / SRTN（短作业/最短剩余时间优先） — 想最小化平均周转，让快的先走
设计依据：可证明 SJF 平均周转最短
缺点：① 需预知服务时间（实际只能估）；② 长作业可能饥饿
抢占式（SRTN）：新到达的若剩余时间更短就抢占

优先级调度 — 体现作业重要性差异
静态优先级：创建时确定不变；动态优先级：随等待时间增加（防饥饿）
缺点：高优持续到达 → 低优饥饿；解决 = 老化 aging

RR 时间片轮转 — 让所有用户体验"同时在用"
设计：每进程分配时间片 q，到期就轮换
q 太小：切换开销大；q 太大：退化为 FCFS
经验值：q ≈ 进程切换时间的 10–100 倍

HRRN 高响应比 — 想兼顾"短作业优先"和"等久了就该跑"
响应比 = (等待+服务)/服务 → 等得越久响应比越高，自动防饥饿
服务时间越短响应比基数越高，短作业仍有优势

多级反馈队列 MLFQ — 不需预知服务时间，自动识别交互/批处理
设计：多个队列，优先级递减；新作业入最高级；时间片用完降一级；I/O 多的留高级
结果：交互短作业自然留高级（快响应）；CPU 密集型沉到低级（不影响他人）
现代 OS（Linux CFS 之前的 O(1) 调度器）均基于此变种

流水线 Pipeline ↑

核心思想：一条指令分成多个阶段，多条指令并行处于不同阶段。

经典 5 段：IF(取指) → ID(译码) → EX(执行) → MEM(访存) → WB(写回)

三种冒险（pipeline hazard）会拖慢流水线：

  • 结构冒险：硬件资源冲突（如同时访存）
  • 数据冒险：后一条依赖前一条结果
  • 控制冒险：分支指令导致 PC 不确定

理想 CPI = 1；有冒险时 CPI > 1

展开：数据冒险的三种缓解

1. 流水线停顿 stall / bubble（最朴素）：插入空指令等前面写回完成。简单但浪费周期。

2. 转发 / 旁路 forwarding（主流）：

  • 前一条算出结果不必等到 WB 写回，从 EX/MEM 阶段直接送回给下一条的 EX 输入
  • 需要硬件多加几条旁路线
  • 消除大多数 RAW（写后读）冒险
无 forwarding：
  ADD R1=R2+R3  IF ID EX MEM WB
  SUB R4=R1-R5     IF ID -- -- ID EX ...  (等3周期)

有 forwarding：
  ADD R1=R2+R3  IF ID EX MEM WB
  SUB R4=R1-R5     IF ID EX MEM WB        (无停顿)
                       ↑ R1从上条EX输出直接转发

3. 加载-使用冒险（load-use）：load 指令在 MEM 阶段才拿到数据，下一条紧跟使用仍需 stall 1 周期；编译器可指令调度把无关指令插入中间消除停顿。

4. 寄存器重命名（高级 CPU）：把虚拟寄存器映射到物理寄存器池，消除 WAR/WAW 假相关。

展开：结构冒险的缓解

方法：

  • 分离指令和数据缓存（哈佛架构）：IF 和 MEM 可同周期不冲突
  • 多端口寄存器堆：允许同周期读写
  • 功能单元复制：多个 ALU 等

缓存与局部性 ↑

层次结构（典型）：

寄存器 ← L1 (≈1 cycle) ← L2 (≈10) ← L3 (≈40) ← 主存 (≈200) ← 磁盘 (10⁶)

局部性原理（虚拟内存和缓存共同的基础）：

  • 时间局部性：刚访问的数据短期内可能再访问 → 缓存留住它
  • 空间局部性：访问某地址时，附近地址也可能被访问 → 整块加载

缓存映射方式：直接映射、组相联、全相联（TLB 通常全相联）

分支预测 ↑

遇到分支时 CPU 猜测走哪条路并继续取指。

  • 预测正确：流水线无延迟
  • 预测错误：流水线清空，损失 10-20 cycles

编程建议：

  • 把可预测的分支（几乎总走某一支）放前面
  • 避免数据相关的随机分支（如随机数组上的 if）
  • 排序后再处理可大幅提速（经典："排序前后的 if 速度差异"）
展开：分支预测的硬件实现层次

1. 静态预测（编译期决定）：

  • 总是预测"不跳转"：简单，准确率约 50%
  • 向后跳转预测"跳"、向前跳转预测"不跳"：利用循环特性，准确率 70%+

2. 动态预测（运行时学习）：

  • 1 位预测器：记最近一次跳没跳，循环边界容易错两次
  • 2 位饱和计数器（最经典）：四状态 SN/WN/WT/ST，连续两次错才改方向；准确率 ~90%
  • 两级自适应：用历史模式索引预测表，可学到"跳-不跳-跳-不跳"周期
  • TAGE/Perceptron：现代 CPU 采用，准确率 95%+

3. 配套结构：

  • BTB（分支目标缓冲）：缓存"分支指令地址 → 目标地址"，预测"跳"时立即知道跳哪
  • RAS（返回地址栈）：专门预测 ret 指令的返回地址

4. 控制冒险的其他缓解：

  • 延迟分支（MIPS）：约定分支后一条指令无论如何都执行（"延迟槽"）
  • 分支预测后投机执行：错则清空流水线和重排序缓冲

"排序后处理快很多"的实验：未排序数据 → 分支随机不可预测 → 预测器准确率约 50%；排序后 → 分支单调 → 准确率 99%；速度差 5-10 倍

代码优化方法 ↑

  • 消除冗余：循环内不变量提出来（loop-invariant motion）
  • 强度削弱：乘除改加减、改位移；i*8 → i<<3
  • 循环展开：减少分支次数，增加指令级并行机会
  • 函数内联：消除调用开销
  • 减少内存访问：用局部变量缓存常用值
  • 消除别名问题：用 restrict 等告诉编译器指针不重叠

缓存友好的代码 ↑

经典对比：二维数组 a[N][N] 求和

// C 是行主序存储
// 慢：列主序遍历，每次跨 N 个元素，缓存命中差
for (j=0; j<N; j++)
  for (i=0; i<N; i++) sum += a[i][j];

// 快：行主序遍历，连续访问，每次缓存命中
for (i=0; i<N; i++)
  for (j=0; j<N; j++) sum += a[i][j];

原则：访问数据的步长越小越好；连续访问利用空间局部性；重复访问利用时间局部性。

矩阵乘法常用分块（blocking/tiling）来提升缓存命中。

乱序执行与超标量 ↑

  • 超标量：CPU 每周期取多条指令并行执行
  • 乱序执行：不按程序顺序，按数据就绪顺序执行
  • 程序员看到的"顺序执行"由重排序缓冲（ROB）保证
  • 影响：单线程内的"并行"也存在，但多线程同步仍需信号量/锁

重要考点要点跳转
★★★银行家算法Need=Max−Alloc；试分配 → 找安全序列概念计算
★★★四必要条件互斥(不可破)、请求保持、不剥夺、循环等待概念

★★预防/避免/检测区别预防：破坏必要条件；避免：银行家；检测：事后概念
★死锁解除剥夺资源、撤销进程、回退概念

死锁四必要条件 ↑

  1. 互斥—— 资源本身性质，无法破坏
  2. 请求与保持—— 破坏：一次申请所有资源
  3. 不剥夺—— 破坏：主动释放或强制剥夺
  4. 循环等待—— 破坏：资源有序分配

处理策略对比 ↑

  • 预防：事前破坏必要条件，能确保不死锁，资源利用率低
  • 避免（银行家）：分配前检查安全性，需要最大需求信息，不会给可能导致死锁的进程分配
  • 检测+解除：事后处理，会分配，发现死锁再解除
  • 忽略（鸵鸟）：现代 OS 常见

银行家算法判断的是"安全状态"，不是"死锁状态"。死锁要用检测算法

银行家算法 ↑

  1. 计算 Need = Max − Allocation
  2. 收到 Request：检查 Request ≤ Need 和 Request ≤ Available
  3. 试分配：Available -= Req; Alloc += Req; Need -= Req
  4. 找安全序列：反复寻找 Need ≤ Available 的进程，让它完成并释放（Available += Alloc）
  5. 找到完整序列 → 安全，正式分配；否则撤销试分配

计算例题：银行家算法（作业二第 7 题） ↑

Available = (1,5,2,0)，4 进程 Allocation 与 Max 见题表。

(1) Need 矩阵：P0(0,0,0,0)、P1(0,0,5,0)、P2(1,0,0,2)、P3(0,6,4,2)

(2) 找安全序列：

  • P0 的 Need=(0,0,0,0) ≤ Avail → 运行，释放后 Avail=(1,5,3,2)
  • P2 的 Need=(1,0,0,2) ≤ Avail → 释放后 Avail=(2,8,8,6)
  • P1 的 Need=(0,0,5,0) ≤ Avail → 释放后 Avail=(3,8,8,6)
  • P3 的 Need=(0,6,4,2) ≤ Avail → 完成

(3) P1 请求 (0,4,2,0)：检查 ≤ Need(0,0,5,0)？ —— 4 > 0，不满足 Need 约束（题目里 Need 是 (0,0,5,0)，第二项是 0，请求 4 超过 Need，应拒绝）

计算技巧：不会死锁的判定 ↑

n 个进程，每个最多需要 k 个某类资源，资源总数 m

m ≥ n × (k − 1) + 1 → 不会死锁

例：5 资源、6 进程每个需 1 → 6×0+1 = 1 ≤ 5 ✓ 不死锁

例：5 资源、3 进程每个需 4 → 3×3+1 = 10 > 5 ✗ 可能死锁

死锁解除 ↑

  • 剥夺资源：抢走部分进程的资源给其他进程
  • 撤销进程：终止某些进程释放资源
  • 回退（rollback）：把进程回退到之前的检查点

死锁时必然有 ≥ 2 个进程处于阻塞态。

分页核心数学 ↑

页大小 = 2^k 字节 → 偏移占 k 位
逻辑地址 m 位 → 页号占 m−k 位
页表项数 = 2^(m−k)
页框数 = 2^(p−k)（p = 物理地址位数）
页表大小 = 项数 × 项字节

计算套路：地址翻译四步 ↑

  1. 页号 = 逻辑地址 >> k
  2. 偏移 = 逻辑地址 & (2^k − 1)
  3. 查页表得页框号
  4. 物理地址 = 页框号 × 页大小 + 偏移

例（作业三第 9.3 题）：页大小 4KB，逻辑 0x00008000，页号 8，装入页框 0x900

页面 2（页号 9）→ 页框号 0x901 → 物理地址 = 0x901 × 0x1000 = 0x00901000

分页地址翻译流程：页号查表 → 拿页框号 → 与偏移拼接为物理地址
展开：四个核心术语区分
  • 页 (Page) / 页面 (Page)：逻辑地址空间的定长块，是程序视角的概念，编号从 0 开始
  • 页号 (Page Number, p)：逻辑地址中标识"是第几页"的高位字段
  • 页框 / 页帧 (Page Frame / Frame, f)：物理内存的定长块，与页大小相同，是硬件视角概念
  • 页表项 (PTE)：页表中的一行，存"页号 → 页框号"的映射，还含权限位、修改位、访问位等

简化口诀：

  • 页是逻辑概念，页框是物理概念，大小相等
  • 页号是地址里的字段，页框号是页表项里的值
  • PTE 是桥梁：从页号查到页框号
  • 偏移在翻译中"透传"：逻辑偏移 = 物理偏移

计算套路：多级页表位数划分 ↑

原则：每级页表本身恰好占一页（这样页表也能用分页管理）

一页能装的项数 = 页大小 / 页表项大小 → 该级索引位数 = log₂(项数)

  • 32 位地址，4KB 页，4B 项 → 一页装 1024 项 = 10 位 → 划分 10+10+12
  • 48 位地址，4KB 页，8B 项 → 一页装 512 项 = 9 位 → 严格需要 9+9+9+9+12（4 级）

表达式（作业三第 9.2 题）：

页目录号 = LA >> 22
页表索引 = (LA >> 12) & 0x3FF
页内偏移 = LA & 0xFFF

计算套路：页表项物理地址 ↑

页表项物理地址 = 页表基址 + 页号 × 项大小

例（作业三第 9.3 题）：页表基址 0x00200000，项大小 4B

  • 页号 8 的页表项 = 0x00200000 + 8×4 = 0x00200020
  • 页号 9 的页表项 = 0x00200000 + 9×4 = 0x00200024

动态分区分配算法 ↑

  • 首次适应 First Fit：空闲链按地址递增排，找第一个够大的（性能综合最好）
  • 最佳适应 Best Fit：找最小的够大的 → 大量小碎片
  • 最坏适应 Worst Fit：找最大的 → 破坏大块

  2. 链接：多个 .o → 可装入模块

    • 运行时动态链接：用到才链（.dll / .so）
  3. 装入：可装入模块 → 内存，有三种方式：

分段地址翻译 ↑

逻辑地址结构：(段号 s, 段内偏移 d)

翻译步骤：

  1. 从段表寄存器找到段表起始地址
  2. 用段号 s 索引段表，得到该段的基址和段长
  3. 检查 d < 段长？否则触发越界中断
  4. 检查权限位（读/写/执行）
  5. 物理地址 = 基址 + d

和分页不同：段内偏移是直接相加，不是拼接（因为段长不固定不能用位运算）

段表项结构 ↑

  • 段长：用于越界检查
  • 权限：R / W / X，按段独立设置
  • 存在位：段是否在内存（虚拟分段）

段页式翻译 ↑

设计目的：兼具分段的逻辑性（共享、保护）和分页的物理高效（无外部碎片）。

逻辑地址结构：(段号 s, 页号 p, 页内偏移 d)

翻译步骤（三次访存）：

  1. 段号 s → 查段表，得到该段的页表起始地址和段长
  2. 检查 (页号, 偏移) 是否越段长
  3. 页号 p → 查该段的页表，得到页框号
  4. 物理地址 = 页框号 × 页大小 + 偏移

因为段大小可变，每段有独立的页表。

不加 TLB 时段页式需要 3 次访存（段表、页表、数据），代价高 → 必须配 TLB

计算例题：段页式翻译 ↑

条件：段号 4 位、页号 8 位、偏移 12 位（页大小 4KB），总 24 位地址

逻辑地址 LA = 0x213ABC（二进制：0010 0001 0011 1010 1011 1100）

偏移 = LA & 0xFFF         = 0xABC   ← 低 12 位
页号 = (LA >> 12) & 0xFF  = 0x13    ← 接下来 8 位
段号 = (LA >> 20) & 0xF   = 0x2     ← 高 4 位

翻译流程：
1. 查段表[0x2] → 得页表起始地址 PT_base + 段长
2. 检查 页号×4KB + 偏移 < 段长 ?
3. 查 PT_base[0x13] → 得页框号 F
4. 物理地址 = (F << 12) | 0xABC

通用套路（从低到高拆位）：

  1. 偏移 = LA & (2^k − 1)，k = 偏移位数
  2. 页号 = (LA >> k) & (2^p − 1)，p = 页号位数
  3. 段号 = LA >> (k + p)

先确定每段位数，再算掩码；不要凭直觉拆 hex 数字

段的共享与保护 ↑

为什么段共享比页共享容易：

  • 段是逻辑单位（一段代码、一个数据结构），整段共享语义清晰
  • 页是物理切片，一个逻辑对象可能跨多页，共享需协调每一页

共享实现：多个进程的段表中相应段表项指向同一物理段。

保护：段表项里的权限位（R/W/X）按段独立设置，可信代码段不可写，数据段不可执行等。

外部碎片与紧凑 ↑

分段会产生外部碎片：内存中散落的小块空闲区，单个都不够用。

紧凑（compaction）：把所有已分配段挪到一边，空闲合并成大块。代价高（要移动数据并更新段表）。

段页式因为段被切成页，物理上不需要连续，消除了外部碎片。

TLB 地址翻译三场景 ↑

  • TLB 命中：TLB 时间 + 内存时间
  • TLB 未命中，页在内存：TLB + 页表 + 内存

缺页处理完后是重新执行整条指令，所以要算两遍地址转换时间

TLB 翻译三场景：命中（绿色快路径）/ TLB miss（查页表）/ 缺页（虚线回路，需重新执行）

计算例题：CLOCK 算法（作业三第 9 题） ↑

4 页框已满，访问位见表，指针指向 2 号页框。

步骤：

  1. 若指针处访问位=0 → 淘汰；新页放入，访问位置 1，指针后移

  3. 一圈内必有 0（被清零的）

DMA 工作流程 ↑

正确顺序：② → ③ → ① → ④

  3. ① DMA 控制器发出中断请求

SPOOLing 假脱机 ↑

UNIX inode 多级索引 ↑

典型设计：12 直接 + 1 一级 + 1 二级 + 1 三级

每个索引块装的指针数 = 块大小 / 指针大小

  • 小文件 → 仅用直接指针，1 次 I/O 即得地址
  • 大文件 → 按需展开多级索引

计算套路：最大文件容量 ↑

令 n = 块大小 / 指针大小（一个索引块装的指针数）
直接覆盖 = 直接指针数 × 块大小
一级覆盖 = n × 块大小
二级覆盖 = n² × 块大小
三级覆盖 = n³ × 块大小
最大容量 = 全部相加

例（作业四第 7 题）：4 直接 + 2 一级 + 1 二级，块 256B，指针 4B

  • n = 256/4 = 64
  • 直接：4 × 256B = 1KB
  • 一级（2 个）：2 × 64 × 256B = 32KB
  • 二级（1 个）：64 × 64 × 256B = 1024KB
  • 总 = 1057KB

数清直接/一级/二级各几个地址项！题目可能给 2 个一级而不是 1 个

计算套路：访问偏移要几次 I/O ↑

  1. 算每级覆盖的块号区间（累积！）
  2. 偏移字节 / 块大小 = 逻辑块号
  3. 判断该块号落在哪一级
  4. I/O 次数 = 该级索引层数 + 1（读数据块）

例（作业四第 6 题）：10 直接+1 一级+1 二级，块 1KB，n=256（inode 已在内存）

  • 直接：块 0–9（覆盖 0–10KB）
  • 一级：块 10–265（覆盖 10KB–266KB）
  • 二级：块 266–65801（覆盖 266KB– ~64MB）

偏移 1234（1.2KB）→ 块 1 → 直接 → 1 次 I/O

偏移 307400（300KB）→ 块 300 → 二级（300 > 265）→ 3 次 I/O（二级索引块+一级索引块+数据块）

(2) 链接分配：顺链找到第 29 条，改两个指针

  • 读 29 次（找位置）+ 写第 29 条新指针 + 写新记录 = 最少约 31 次
  • 最大文件长度：若 FCB 只存起始块号，理论无上限（直到磁盘满）

打开文件时 OS 做什么 ↑

主要工作：把文件控制块（FCB / inode）读入内存的打开文件表

  • 不会预读文件内容
  • 不会修改 FCB 中的读写权限位
  • 返回的是文件描述符（打开文件表的索引），不是数据缓冲区指针

文件访问加速 ↑

下列都能提速：

  • 提前读：预测下次访问，先读到缓冲
  • 连续分配簇：减少寻道
  • 延迟写：合并多次写
  • 磁盘高速缓存：内存缓存常用块

目录 ↑

主要功能：实现文件名到文件地址（FCB/inode）的映射

UNIX 设计：目录项 = (文件名, inode 号)，元数据全在 inode 里 → 硬链接容易实现

逻辑结构 vs 物理结构 ↑

  • 逻辑结构：用户看到的文件组织形式（流式、记录式）
  • 物理结构：磁盘上的存储方式（连续/链接/索引）

目录结构演进 ↑

结构特点缺点
单级目录全系统一张表重名冲突、查找慢
两级目录主目录+用户子目录用户间不便共享
树形目录多级嵌套（DOS/Windows）一文件一父，不支持共享
无环图目录多父（UNIX 硬链接）删除需引用计数
通用图目录允许循环难判垃圾，需 gc

UNIX 用无环图：硬链接实现一文件多父；同一 inode 被多个目录项指向。

空闲空间管理（四种方法） ↑

方法结构特点
位图每位代表一块（0空闲/1占用）紧凑，找连续块快
空闲链表空闲块串成链不占额外空间，找连续块慢
空闲表每项 (起始, 长度)适合连续分配方式
成组链接（UNIX）每 100 块组成一组，组尾记下一组地址兼顾空间和速度

成组链接的精妙：只需在内存里保留一个组的信息（≤ 100 个块号），分配回收都快；组用完时换下一组到内存。

FAT 文件系统 ↑

核心思想：把所有"下一块"指针集中到一张 FAT 表里，整张表常驻内存。

FAT 表项的值含义：

  • 正常值 → 下一块号
  • 特殊值 → EOF（文件结束）、坏块、空闲块

优点：常驻内存 → 随机访问快（不必读盘走链）

缺点：FAT 本身占大量内存；表项位数限制了最大簇数

变种：FAT12（软盘）/ FAT16 / FAT32（28 位有效位）/ exFAT

计算例题：FAT 分区大小（2024 真题第 19 题） ↑

题：块大小 4KB，FAT 表项 32 位（1 位已占用、1 位坏块，其余 30 位表块号），求最大分区和 4GB 分区的 FAT 表长。

最大分区：
  可表示块号 = 2^30
  最大分区 = 2^30 × 4KB = 2^30 × 2^12 B = 2^42 B = 4TB

4GB 分区的 FAT 表长：
  块数 = 4GB / 4KB = 2^20
  FAT 表 = 2^20 × 4B = 4MB

注意区分"FAT 表项位数"和"有效块号位数"（要扣掉保留位）

EXT2 文件系统结构 ↑

Linux 经典文件系统，被 EXT3/4 继承。整个分区分成块组（Block Group）。

每个块组的结构（自前向后）：

  1. 超级块（Superblock）：文件系统全局信息（块大小、总块数、inode 数等），每组都有副本以防损坏
  2. 块组描述符表：所有块组的元数据位置
  3. 数据块位图：本组数据块的空闲情况（1 块）
  4. inode 位图：本组 inode 的空闲情况（1 块）
  5. inode 表：本组所有 inode 集中存放
  6. 数据块区：剩余空间存数据

为什么分块组：

  • 同一文件的 inode 和数据块尽量放在同一块组 → 减少寻道
  • 位图分组 → 单个位图块就能管理一整组
  • 超级块多副本 → 提高可靠性

典型 inode 结构：12 直接 + 1 一级 + 1 二级 + 1 三级（共 15 个指针）

EXT2 相比 FAT 的优势：随机访问快（inode 多级索引）、并行友好（块组）、可靠（多副本）

硬链接 vs 软链接 ↑

硬链接 hard link软链接 symlink
本质多个目录项指向同一 inode独立文件，内容是目标路径

跨文件系统❌ 不能✅ 可以

UNIX 删除文件：实际是删除目录项 + inode 链接计数 -1；只有计数归零才真正释放

① 银行家算法 · 死锁避免

3/3 真题出现

题目特征：给出 Allocation 矩阵、Max 矩阵、Available 向量，问安全状态 + 新请求能否满足。

标准七步法：

  1. 列 Need 矩阵：Need[i] = Max[i] − Allocation[i]，逐元素相减
  2. 检查 Available：写下当前可用资源向量
  3. 找第一个可执行进程：扫描所有进程，找 Need[i] ≤ Available 的（每个分量都满足）
  4. 执行并释放：该进程加入安全序列，Available += Allocation[i]
  5. 重复步骤 3-4：直到所有进程入列 → 安全；找不到 → 不安全
  6. 处理新请求：检查 Request ≤ Need[i] 且 Request ≤ Available，否则拒绝
  7. 试分配后重判：Available -= Req; Alloc += Req; Need -= Req，再走步骤 3-5

真题2018 秋 / 2024 秋第 3 题

4 进程 4 类资源，Available = (1,5,2,0)：

Need 矩阵：
  P0: (0,0,0,0)   ← 已无需求
  P1: (0,0,5,0)
  P2: (1,0,0,2)
  P3: (0,6,4,2)

寻找安全序列：
  Avail=(1,5,2,0)  → P0 Need 全 0，可执行 → 释放 (0,0,1,2)
  Avail=(1,5,3,2)  → P2 Need=(1,0,0,2) ≤ Avail → 释放 (1,3,5,4)
  Avail=(2,8,8,6)  → P1 Need=(0,0,5,0) ≤ Avail → 释放 (1,0,0,0)
  Avail=(3,8,8,6)  → P3 Need=(0,6,4,2) ≤ Avail → 完成
  
安全序列：P0 → P2 → P1 → P3 ✓

新请求 P1 Req=(0,4,2,0)：
  检查 (0,4,2,0) ≤ Need_P1=(0,0,5,0)？ 4 > 0 ✗ 拒绝

常见失分点：① 忘了 Need = Max−Alloc 这一步；② 比较时漏掉某个资源类；③ 试分配后忘记更新三个矩阵；④ 找到一个安全序列就够了，不必找全部

② 多级页表 / 段页式地址翻译

3/3 真题出现

题目特征：给地址位数、页大小、页表项大小，求页大小 / 页表占用 / 物理地址 / 页表项位置。

标准五步法（一级/二级页表）：

  1. 算偏移位数：k = log₂(页大小)。如 4KB 页 → k=12
  2. 拆地址：[页目录号][页号][页内偏移]，按题目给的位宽划分
  3. 用 LA 算各字段： 
    页目录号 = LA >> (二级位数+偏移位数)
页表索引 = (LA >> 偏移位数) & ((1 << 二级位数) − 1)
偏移     = LA & ((1 << 偏移位数) − 1)
  4. 查页表得页框号：页表项地址 = 页表基址 + 页号 × 项大小
  5. 拼物理地址：PA = 页框号 × 页大小 + 偏移

段页式额外步骤：先用段号查段表得页表基址，再走上面流程。

真题2018 / 2024 第 4 题（32 位地址，4KB 页，4B 项，二级页表 10+10+12）

代码段起始逻辑地址 0x00008000，装到物理地址 0x00900000，页表存放在 0x00200000：

(1) 页大小：2^12 = 4KB
    一级页表占用：2^20 × 4B = 4MB

(2) 给逻辑地址 LA：
    页目录号 = LA >> 22
    页表索引 = (LA >> 12) & 0x3FF
    偏移     = LA & 0xFFF

(3) 0x00008000 → 页号 8 = 0x00008000 >> 12
    8KB 占 2 页 → 页号 8 和 9
    
    页表项地址：
      页号 8 → 0x00200000 + 8×4 = 0x00200020
      页号 9 → 0x00200000 + 9×4 = 0x00200024
    
    页框号：
      0x00900000 / 0x1000 = 0x900 (页框 1)
      0x00901000 / 0x1000 = 0x901 (页框 2)
    
    代码页面 2 起始物理地址 = 0x901 × 0x1000 = 0x00901000

真题2025 秋第 4 题（48 位地址，4KB 页，8B 项）

判断需要几级页表：

偏移：12 位
剩余：48 − 12 = 36 位用于页号

一级页表项数 = 2^36，占空间 = 2^36 × 8B = 512GB ✗ 不可行
→ 必须用多级

每级页表占一页 → 一页装 4KB/8B = 512 项 = 9 位
36 / 9 = 4 → 需要 4 级页表

平均访问时间（一级 + 98% TLB 命中）：
  EAT = 10 + 0.98 × 100 + 0.02 × (100 + 100) = 112ns

常见失分点：① 位运算搞错（用 0x3FF 而不是 0xFFF）；② 页表项地址忘了乘项大小；③ 算页表占用空间时漏算项数；④ 段页式访存次数算错（无 TLB 是 3 次）

③ 多级索引文件系统

3/3 真题出现

题目特征：给 inode/FCB 地址项数（直接 + 一级 + 二级 + 三级）、块大小、指针大小，求最大文件 / 某偏移的 I/O 次数 / 实际占用空间。

标准四步法：

  1. 算一个索引块装多少指针：n = 块大小 / 指针大小
  2. 累加各级覆盖容量： 
    直接 = 直接数 × 块大小
一级 = 一级数 × n × 块大小
二级 = 二级数 × n² × 块大小
三级 = 三级数 × n³ × 块大小
最大文件 = 上述全部相加
  3. 定位某偏移： 
    逻辑块号 = 字节偏移 ÷ 块大小
判断该块号落在哪一级（按累积区间）
  4. I/O 次数：= 索引层数 + 1（最后 +1 是读数据块本身；inode 已在内存不算）

真题2018 秋第 6 题（13 项：10 直 + 1 一 + 1 二 + 1 三，块 1KB，指针 3B）

n = 1024 / 3 = 341（向下取整）
直接：10 × 1KB = 10KB
一级：341 × 1KB ≈ 341KB
二级：341² × 1KB ≈ 116MB
三级：341³ × 1KB ≈ 39GB
最大 ≈ 39GB

1GB 文件实际占用（包含索引块）：
1GB = 2^30 字节，需 2^20 个 1KB 数据块
全部在三级索引下：
  数据块：2^20 个
  二级索引块：⌈2^20 / 341⌉ ≈ 3074 个
  一级索引块：⌈3074 / 341⌉ ≈ 10 个
  三级索引块：1 个
总占用 ≈ 1GB + 3MB（索引开销）

真题2024 秋第 6 题（13 项：10 直+1 一+1 二+1 三，块 512B，指针 3B）

n = 512 / 3 = 170
直接：10 × 512B = 5KB
一级：170 × 512B = 85KB
二级：170² × 512B ≈ 14MB
三级：170³ × 512B ≈ 2.4GB

字节偏移 5000 → 块号 = 5000/512 = 9 → 直接索引
  访问 = 1 次（数据块）

字节偏移 150000 → 块号 = 150000/512 = 292
  10 ≤ 292 < 10+170=180？ 292 > 180 → 二级
  访问 = 3 次（二级索引 + 一级索引 + 数据）

最少 I/O（直接、且块在 inode 中）= 1 次
最多 I/O（三级、块在外存）= 4 次

常见失分点：① 数错地址项数（10 还是 12 个直接？）；② 区间累积错误（一级覆盖是 [直接+1, 直接+n] 不是 [0,n]）；③ 算 I/O 时忘了 inode 是否在内存；④ 实际占用没算索引块开销

④ PV 同步设计

3/3 真题出现

题目特征：描述一个并发场景（生产消费、读写、爸爸-儿女吃水果等），要求用 PV 操作实现。

标准五步法：

  1. 识别角色和动作：列出有哪些类型的进程，每个进程做什么
  2. 识别共享资源：找需要互斥访问的变量/数据结构 → 配 mutex = 1
  3. 识别等待关系：谁等谁？每一种等待 → 一个同步信号量，初值通常 0
  4. 识别资源数量：有限缓冲区/座位 → 信号量初值 = 数量
  5. 写代码遵守模板： 
    // 互斥包临界区
P(mutex); ...; V(mutex);

// 同步：前驱末尾 V，后继开头 P
// 资源池：用前 P，用后 V

// 永远不要持有 mutex 时去 P 阻塞信号量
错误：P(mutex); P(empty); ... 
正确：P(empty); P(mutex); ... V(mutex); V(full);

真题2025 秋第 1 题：爸爸放水果，女儿吃苹果，儿子吃橘子，盘子最多 1 个

// 三个同步信号量
semaphore plate  = 1;  // 盘子空位（资源池）
semaphore apple  = 0;  // 苹果就绪事件
semaphore orange = 0;  // 橘子就绪事件

Father:                   Daughter:              Son:
while(true) {             while(true) {          while(true) {
  P(plate);                 P(apple);              P(orange);
  随机放苹果或橘子;          取苹果吃;              取橘子吃;
  if (放苹果) V(apple);     V(plate);              V(plate);
  else        V(orange);  }                      }
}

分析：plate=1 保证盘子互斥+容量；apple/orange=0 用于通知"对应水果已就绪"；女儿/儿子吃完才 V(plate) 给爸爸放新水果

真题2024 秋第 1 题：生产者-消费者，缓冲区 1000，"一消费者连续取 10 件其他才能取"

semaphore mutex = 1;       // 缓冲区互斥
semaphore empty = 1000;    // 空位
semaphore full  = 0;       // 产品数
semaphore exclusive = 1;   // 消费者独占权

Producer:
while(true) {
  produce();
  P(empty); P(mutex);
  put(item);
  V(mutex); V(full);
}

Consumer:
while(true) {
  P(exclusive);            // 抢到独占权
  for (int i=0; i<10; i++) {
    P(full); P(mutex);
    take(item);
    V(mutex); V(empty);
    consume();
  }
  V(exclusive);            // 取完 10 件释放独占权
}

真题2018 秋第 5 题：银行 4 窗口 + 10 座位，叫号系统

semaphore mutex_machine = 1;  // 取号机互斥
semaphore seats         = 10; // 座位数
semaphore customers     = 0;  // 等待的客人数（叫号信号）
semaphore served        = 0;  // 服务就绪（客人等叫号）

Customer:
P(seats);                  // 抢座位

V(customers);              // 通知营业员

接受服务;

常见失分点：① mutex 和阻塞信号量顺序错（先 P 资源再 P mutex）；② 漏掉某种等待关系；③ 共享变量没用 mutex 保护；④ V 操作位置错（应该在改完状态后立刻 V）

⑤ 调度算法 / 甘特图

3/3 真题出现

题目特征：给若干进程的到达时间+服务时间，求 FCFS / SJF / RR / HRRN 等算法下的甘特图和平均周转时间。

标准四步法：

  1. 列时间事件点：所有进程的到达时刻 + 当前运行进程的结束时刻 + 时间片到期点（RR）
  2. 每个事件点重新决策：按算法规则选下一个运行的进程
    • FCFS：选到达最早的就绪进程
    • SJF（非抢占）：选已到达中服务时间最短的
    • SJF/SRTN（抢占）：选剩余时间最短的
    • RR：按队列顺序，每个跑一个时间片
    • HRRN：每次空闲时算所有等待进程的响应比，选最高
  3. 画甘特图：时间轴上标出每段运行的进程和起止时刻
  4. 算周转时间：完成 − 到达 → 求平均

真题2024 秋第 2 题：P0(0,8)、P1(0.4,4)、P2(1.0,1)，非抢占式

FCFS：按到达顺序
  甘特图：P0[0,8] | P1[8,12] | P2[12,13]
  周转：P0=8, P1=11.6, P2=12 → 平均 = 10.53

SJF：P0 已开始（非抢占），跑完后从已到达中选最短
  甘特图：P0[0,8] | P2[8,9] | P1[9,13]
  周转：P0=8, P1=12.6, P2=8 → 平均 = 9.53

Future-knowledge：CPU 空闲 1 单位，等 P1、P2 都到了再选最短
  甘特图：idle[0,1] | P2[1,2] | P1[2,6] | P0[6,14]
  周转：P0=14, P1=5.6, P2=1 → 平均 = 6.87

真题2025 秋第 3 题：同时到达 5 进程，时间片=1

FCFS（按到达顺序 P0,P1,P2,P3,P4）：
  P0[0,5] | P1[5,6] | P2[6,8] | P3[8,9] | P4[9,12]
  周转：5,6,8,9,12 → 平均 = 8

SJF（非抢占，全已到达，按服务时间排序：P1=1,P3=1,P2=2,P4=3,P0=5）：
  P1[0,1] | P3[1,2] | P2[2,4] | P4[4,7] | P0[7,12]

★ 核心公式速查（考前 10 分钟）

调度算法
  • 周转 = 完成 − 到达
  • 带权周转 = 周转 / 服务
  • 等待 = 周转 − 服务
  • 响应比 = (等待+服务) / 服务
  • CPU 利用率 = CPU 忙 / 总时间
死锁判定
  • n 进程每个需 k 资源、共 m 资源
  • m ≥ n(k−1) + 1 → 不死锁
  • Need = Max − Allocation
  • 找安全序列：Need ≤ Available
地址翻译
  • 页大小 = 2^k → 偏移 k 位
  • 页号 = LA >> k
  • 偏移 = LA & (2^k − 1)
  • 物理地址 = 页框号 × 页大小 + 偏移
  • 页表项地址 = 基址 + 页号 × 项大小
多级页表
  • 每级项数 = 页大小 / 项大小
  • 4KB 页 + 4B 项 → 每级 10 位
  • 4KB 页 + 8B 项 → 每级 9 位
TLB 与缺页
  • 命中：TLB + 内存
  • 未命中：TLB + 页表 + 内存
  • 缺页：TLB + 页表 + 缺页处理 + 重新执行
磁盘
  • 容量 = 盘面 × 柱面 × 扇区 × 扇区大小
  • 访问时间 = 寻道 + 旋转(半圈) + 传输
  • 7200 RPM → 旋转延迟 4.17ms
缓冲区
  • 单缓冲：max(C, T) + M
  • 双缓冲：max(C, T+M)
inode 容量（n = 块/指针）
  • 直接：直接数 × 块
  • 一级：n × 块（每个一级项）
  • 二级：n² × 块
  • 三级：n³ × 块
  • I/O 次数 = 索引层数 + 1
信号量
  • 互斥：初值 1，P/V 同进程
  • 同步：初值 0，P/V 跨进程
  • 资源池：初值 = 资源数
  • S>0：可用数；S<0：|S| 等待数
启动顺序
  • 引导：ROM → MBR → 分区 → OS
  • 制盘：物理格式化 → 分区 → 逻辑格式化 → 装 OS
  • 扇区划分 ← 物理格式化
  • 根目录建立 ← 逻辑格式化
异常分类
  • 中断：异步、外部
  • 陷阱：同步、故意（系统调用）
  • 故障：同步、可恢复（缺页）
  • 终止：同步、不可恢复
分段 / 段页式
  • 分段：物理 = 基址 + 偏移（先检查越界）
  • 段页式：(s,p,d) → 段表 → 页表 → 物理
  • 段页式访存 3 次（需 TLB）
  • 段共享易：段是逻辑单位
现代处理器
  • 5 段流水线：IF/ID/EX/MEM/WB
  • 三冒险：结构、数据、控制
  • 分支预测错 → 流水线清空
  • 缓存友好：行主序、步长 1
  • 局部性：时间 + 空间
