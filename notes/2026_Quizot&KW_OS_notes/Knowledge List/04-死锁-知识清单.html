一、死锁的引入

1.1 死锁的概念 必考

死锁（Deadlock）：多个进程（线程）因循环等待资源而造成无法执行的现象。

  • 死锁会造成进程无法执行，系统资源无法释放，极大浪费。
  • 经典现象：进程A占有资源1等待资源2，进程B占有资源2等待资源1 --> 循环等待 --> 死锁。

信号量引发死锁示例（deadlock.c）：

// Tid[0]          Tid[1]:
// P(s0)           P(s1)
// P(s1)  <-死锁-> P(s0)
// cnt++           cnt++
// V(s0)           V(s1)
// V(s1)           V(s0)

两个线程以相反顺序申请两个互斥信号量，必然进入死锁区。

易错：有环路不一定死锁（当资源有多个实例时，环路只是必要条件而非充分条件）。

1.2 进度图分析死锁 重点

死锁区（Deadlock Region）：进度图中，任何进入该区域的执行轨迹都会最终到达死锁状态，无限等待某信号量变为非0。

  • 死锁经常是非确定的（nondeterministic）：死锁的产生与调度顺序有关。
  • 不同执行轨迹可能避开或进入死锁区。

二、死锁的四个必要条件

2.1 四个必要条件 必考

产生死锁必须同时满足以下四个条件：

条件含义
互斥使用（Mutual Exclusion） 至少有一个资源一次只能被一个进程占用
不可抢占（No Preemption） 资源只能由持有者自愿释放，不能强行剥夺
持有并等待（Hold and Wait） 进程已持有资源，又在等待其他资源
循环等待（Circular Wait） 在资源分配图中存在一个环路
易错：四个条件是必要条件而非充分条件，即同时满足四个条件不一定死锁（资源有多实例时）；但死锁一定满足这四个条件。

2.2 资源分配图（RAG） 重点

资源分配图模型：进程集合 {P1,P2,...,Pn}，资源类型集合 {R1,R2,...,Rm}，资源 Ri 有 Wi 个实例（黑点表示）。

  • 资源请求边：有向边 Pi --> Rj（进程请求资源）
  • 资源分配边：有向边 Ri --> Pk（资源已分配给进程）

环路与死锁的关系：

  • 若每种资源只有一个实例：存在环路 <=> 死锁
  • 若某种资源有多个实例：存在环路是死锁的必要非充分条件

三、死锁处理方法

3.1 死锁预防 必考

思想：破坏死锁四个必要条件之一，从根本上防止死锁发生（静态策略）。

破坏条件方法缺点
破坏互斥使用 允许资源共享 资源固有特性，通常无法破除（如打印机）
破坏不可抢占 持有资源的进程申请新资源被拒时，释放已持有资源 只对可保存/恢复状态的资源有效（CPU、内存），对打印机等不适用
破坏持有并等待 进程执行前一次性申请所有资源 需预知未来（编程困难）；资源利用率低
破坏循环等待 资源有序分配：对资源类型排序，申请必须按序进行 编程别扭；为申请低序号资源需先释放高序号资源，效率降低
易错：银行家算法是死锁避免，不是死锁预防。资源有序分配才是破坏循环等待（死锁预防）。

3.2 死锁避免（银行家算法） 必考

思想：动态检测每次资源请求，若分配后系统仍处于安全状态则允许，否则拒绝（摸石头过河）。

核心概念

  • 安全状态：存在一个可完成的执行序列 <P1,...,Pn>，每个进程所需资源 ≤ 剩余资源 + 前面已完成进程释放的资源。
  • 安全序列：上述可完成的执行序列。
  • 安全状态 --> 不死锁；不安全状态不一定死锁，但可能死锁。

银行家算法数据结构

  • Available[m]：各类资源当前可用总量
  • Allocation[n][m]：当前已分配给每个进程的各类资源量
  • Need[n][m]：每个进程还需要的各类资源量
  • Work[m]：算法中动态追踪可用资源（初始 = Available）
  • Finish[n]：进程是否能完成

安全性检测算法（时间复杂度 O(mn²)）

① Work = Available; Finish[1..n] = false
② 找未完成且 Need[i] <= Work 的进程 Pi
③ Work = Work + Allocation[i]; Finish[i] = true; GOTO ②
④ 若所有 Finish[i]=true，则安全；否则不安全（死锁）

资源请求算法

若 Request[i] > Need[i]  --> 错误
若 Request[i] > Available --> 等待
试分配：Available -= Request; Allocation[i] += Request; Need[i] -= Request
调用安全性算法：若不安全 --> 回滚拒绝；若安全 --> 分配成功

银行家算法经典示例

5进程3资源（A/B/C），初始 Available=(3,3,2)，安全序列为 <P1,P3,P4,P0,P2>。

易错：安全序列不唯一；不安全状态不等于已死锁；每次资源请求都需重新运行安全性算法，开销大。

银行家算法的缺点

  • 要求进程声明最大资源需求，对程序员要求高
  • 安全性检测 O(mn²) 过于复杂，每次请求都执行效率低
  • 资源利用率大幅降低（可用资源因不安全而不被分配）

3.3 死锁检测与恢复 重点

思想：只要可用资源足够则先分配，定期运行检测算法，发现死锁再处理（掉坑爬出）。

死锁检测算法（改进版银行家）

  • 与安全性算法区别：Finish[i] 初始化时，若 Allocation[i]==0 则 Finish[i]=true（无分配资源的进程视为已完成）
  • 使用 Request[n][m] 替代 Need[n][m]（当前实际请求量）
  • 检测到 Finish[i]=false 的进程 --> 死锁进程组
  • 可通过资源分配图（RAG）中的环路检测（如强连通分量算法）辅助判断

死锁恢复方法

  • 方法一：终止进程
    • 终止所有死锁进程（简单但代价大）
    • 逐个终止代价最小的进程直到解除死锁（考虑：优先级、已执行时间、资源类型、剩余需求等）
  • 方法二：剥夺资源（进程需要回滚 Rollback）
    • 选择牺牲品：代价最小的进程
    • 回滚：将进程回滚到安全状态（部分回滚 or 完全回滚）
    • 防止饥饿：回滚次数计入代价因素

3.4 死锁忽略（鸵鸟算法） 常考

思想：对死锁不做任何处理，死锁发生时手动干预（重启）。

  • UNIX、Windows 等大多数通用操作系统均采用此策略
  • 原因：死锁出现非确定，避免死锁付出的代价过大且往往毫无意义

3.5 四种方法对比 重点

方法思路代价/缺点实际使用
死锁预防 破坏必要条件（静态） 引入不合理因素，效率低 嵌入式系统
死锁避免 银行家算法动态检测 O(mn²)，效率过低 理论研究
死锁检测+恢复 允许死锁发生，发现后恢复 实现较难，恢复代价大 部分数据库
死锁忽略 不处理，重启解决 用户体验差 UNIX/Windows（最常用）
