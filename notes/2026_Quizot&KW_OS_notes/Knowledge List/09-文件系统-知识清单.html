1. 文件与目录结构

1.1 文件的基本概念 重点

文件：磁盘上有名字的字节序列（named bytes on disk）。用户眼中是连续字符流，文件系统眼中是磁盘块集合。

文件系统的核心任务：将（文件名, 偏移）映射到磁盘块地址，目标是磁盘访问次数尽量少、空间开销最小。

文件的物理结构：信息在物理存储器上的存储方式，分三种（见1.2）。
文件的逻辑结构：用户对信息进行逻辑组织形成的结构，向用户提供逻辑清晰的访问形式。

1.2 盘块的三种分配方式 必考

方式FCB中存储优点缺点
连续分配起始盘块号 + 块数简单、支持随机访问外部碎片，扩充难
链式分配第一块块号（每块末存下一块号）利用率高，易扩充只能顺序访问，效率低
索引分配索引块块号支持直接访问，无外部碎片索引块本身占空间；需解决索引块大小问题
易错：链式分配中每块末尾存放"下一块块号"；FAT文件系统将链表集中存放为文件分配表，属于显式链接，优于隐式链接。

1.3 UNIX多级索引（组合方案） 必考

inode中使用组合索引：前12项直接索引（直接寻址），第13项一级间接，第14项二级间接，第15项三级间接。

以1KB块大小、4B块号为例：每块含256条索引。

  • 12个直接块：12 KB
  • 一级间接：256 x 1KB = 256 KB
  • 二级间接：256 x 256 x 1KB = 64 MB
  • 三级间接：256 x 256 x 256 x 1KB = 16 GB
易错：索引块本身存放在数据块区，不占inode空间；inode设计固定大小以便快速查找。

1.4 树状目录与路径解析 必考

文件系统用树状（k叉树）目录组织文件，解决查找困难问题。

目录是一种特殊文件，其数据内容是<文件名, FCB指针（inode号）>的映射表，而非完整FCB。原因：目录解析频繁，FCB尺寸大，存完整FCB会使目录文件很大、效率低。

路径解析示例（/my/data/a）：

  1. 从根目录FCB读取根目录数据块，查找名为"my"的目录项，得其inode号
  2. 读取my目录的FCB及数据块，查找"data"
  3. 读取data目录的FCB及数据块，查找"a"，返回文件a的FCB

根目录FCB存放在磁盘固定位置，OS初始化时读入内存常驻。

2. 文件系统的实现

2.1 分区结构（UNIX分区） 必考

典型UNIX分区布局（从低地址到高地址）：

区域内容
引导块引导OS信息；无OS则为空
超级块（Super Block）分区块数、块大小、空闲块数量/指针、空闲FCB数量/指针等
索引节点数组（inode数组）所有文件的文件头连续存放；UNIX root目录inode号为2
数据块文件内容、目录内容
易错：超级块在UFS中称为超级块，在NTFS中存于主控文件表（MFT）；EXT2中每个块组都有超级块备份。

2.2 空闲盘块管理 重点

  • 空闲位图（位向量）：每一位对应一个盘块，0=空闲，1=占用。可快速分配连续块组，但位图可能很大（1GB/1KB = 1M位 = 128KB位图）。
  • 空闲链表：将空闲盘块用指针链接。分配单块高效，分配多块需遍历。幸运的是大多数申请只需一两个空闲块。

2.3 磁盘缓存（Buffer Cache） 重点

在内存中缓存磁盘盘块，利用局部性减少磁盘访问。访问流程：

  1. 首先查缓存（Hash表组织）
  2. 缓存未命中 --> 访问文件系统（磁盘）
  3. 将盘块放入缓存（替换算法：LRU）

写策略：延迟写（提高效率，断电丢失风险）或直接写（同步写）。fsync()系统调用可绕过缓冲机制直接写磁盘。

2.4 文件保护 常考

文件权限（r/w/x）存放在FCB/inode中。权限分三类：owner(uid)、group(gid)、other。

访问控制流程：进程发起文件访问 --> 检查PCB中的uid/gid --> 对比inode中的权限位 --> 允许/拒绝。

权限字符串示例：drwxrwxr-x root staff test/（d=目录，rwx各代表读写执行）。

2.5 容错：RAID 常考

RAID（Redundant Arrays of Inexpensive Disks）：通过冗余提高可靠性。

  • RAID1（镜像）：每次写两个磁盘，读时若磁盘A出错则读磁盘B，磁盘利用率50%。
  • RAID5+：校验数据（XOR）分布在多个盘上，任意一个磁盘故障可从其他盘恢复，磁盘利用率高于RAID1。
易错：RAID5校验块分散到各磁盘的目的是避免校验块所在磁盘成为写操作瓶颈。

3. EXT2文件系统

3.1 EXT2磁盘组织 必考

EXT2将分区划分为若干块组（Block Group），每个块组包含：

  1. 超级块：整个分区文件系统信息（多个块组中冗余备份，仅块组0必须读入内存）
  2. 块组描述符GDT：描述所有块组属性，每个描述符32字节
  3. 数据块位图：标记当前块组内各数据块是否空闲（1个4KB位图块可标记32K个块，128MB/块组）
  4. inode位图：标记各inode是否使用（1个4KB位图块标记32K个inode）
  5. inode表：当前块组所有inode连续存放（每个inode固定128B）
  6. 数据块：文件/目录数据
易错：EXT2的inode数量在格式化时固定，因此每个块组的最大文件数受限。

3.2 inode结构 必考

inode（索引节点）包含：文件元数据（大小、权限、时间戳等）+ i_block[]（12直接+1一级间接+1二级间接+1三级间接）。

inode不包含文件名！文件名只存在于目录项（ext2_dir_entry）中，这是EXT2的独特性，支持硬链接（多个文件名指向同一inode）。

inode缓存（打开文件的inode放入内存）与页缓存（文件数据页）是两套不同的缓存机制。

3.3 EXT2目录项 重点

目录是一种特殊文件，其数据块是ext2_dir_entry结构的列表，每项包含：inode号、文件名长度、记录总长度、文件类型、文件名。

文件类型取值：0=未知，1=普通文件，2=目录，3=字符设备，4=块设备，5=FIFO，6=套接字，7=符号链接。

3.4 EXT2多级索引计算 必考

典型考研计算（块大小1KB，块号4B，每块256条索引）：

最大文件 = 12 + 256 + 256² + 256³ 块 ≈ 16GB

块大小4KB（1024条索引/块）：最大文件 = 12×4KB + 1024×4KB + 1024²×4KB + 1024³×4KB ≈ 4TB

3.5 EXT4 Extent优化 常考

Extent：用B+树替代多级索引，将连续物理块记录为一条区段（起始块号+连续块数），大幅减少间接块数量，优化大文件访问。

B+树特点：宽泛矮胖（一个节点占一个存储块）；所有数据在叶子节点；按块号有序；减少磁盘I/O次数。

4. FAT文件系统

4.1 FAT卷结构 必考

FAT（File Allocation Table）卷布局（从低到高）：

区域说明
引导区引导代码、FAT格式信息、扇区大小、簇大小等
FAT1文件分配表（簇链）
FAT2FAT1的镜像备份
根目录FAT12/16预留256项；FAT32无数量限制
数据区文件/目录数据

簇（Cluster） = n个扇区，是FAT文件系统的基本分配单位。FAT16最大簇数2^16 = 64K，限制单个文件最大4GB。

4.2 FAT表 必考

FAT表每项记录对应簇的"下一个簇号"，构成簇链（显式链接）。

  • 0x0000：该簇空闲
  • 0x0002~0xFFEF：下一个簇的簇号（文件链指针）
  • 0xFFF0~0xFFFF：特殊状态（文件结束/坏簇/保留）

FAT32每项4字节，最多限定使用2^28个簇（0.25G个），规定簇大小4~32KB，支持卷大小最大1TB~8TB。

易错：FAT2是FAT1的镜像备份，目的是防止FAT1损坏导致整卷瘫痪。

4.3 FAT目录项 重点

FAT32目录项固定32字节，包含：文件名、属性字节（r/w/x等）、时间戳、起始簇号、文件大小。

文件访问过程（读 /aa/bb/ccc.dat）：

  1. 查根目录，找目录项名="aa"，得到aa的首簇号
  2. 顺着FAT簇链读aa目录数据，找目录项名="bb"，得bb首簇号
  3. 顺着FAT簇链读bb目录数据，找文件名="ccc.dat"，得首簇号
  4. 顺着FAT簇链读取文件ccc.dat所有数据块

4.4 FAT vs EXT2比较 必考

特性FAT32目录项EXT2目录项
本质完整的FCB（包含所有文件元数据）仅是文件名到inode的映射（链接）
文件元数据自包含（大小、时间戳、起始簇号等）均在inode中，目录项不包含
与文件实体关系强耦合：删除目录项 = 删除文件弱耦合：支持硬链接（多个名字指向同一inode）
分配方式链表（显式链接），整个分区内分配索引（多级/Extent），首先在块组内分配

5. 日志文件系统与性能优化

5.1 文件系统日志（Journal） 常考

传统文件系统在断电/崩溃时会出现不一致（如链接断裂、目录项与inode不同步）。

Journal（日志）解决方案：写数据前先写到日志区，保证写入的原子性。

Undo型日志崩溃恢复：检查是否有未提交的COMMIT --> 定位不一致数据 --> UNDO回滚到崩溃前状态。

5.2 日志型文件系统LFS 了解

LFS（Log-Structured Filesystem）：将所有写入顺序追加到日志末尾，消除传统随机写的磁盘寻道开销。

问题：数据散布全盘，需要内存索引（二叉搜索树）高效定位。日志写满时进行垃圾回收（辨别无效数据 --> 搬运有效数据 --> 释放空间）。

F2FS（Flash Friendly Filesystem）：针对NAND Flash的日志型文件系统，冷热数据分离，减少写入放大，随机写性能比EXT4提升约2倍。

5.3 EXT4 I/O性能优化 常考

  • 写缓冲（延迟写）：数据先进写缓冲区，5~30秒后批量写盘；fsync()可强制立即写盘。
  • 延迟块分配：磁盘块分配延迟到数据真正写入时（页面刷新时），降低文件碎片化，避免为临时内容分配磁盘块。
  • Extent B+树：连续物理块用区段表示，减少间接块访问次数（见3.5节）。
  • inode/目录局部性：同一目录中的文件inode集中在同一柱面组；根目录FCB常驻内存；当前目录FCB可驻内存。
