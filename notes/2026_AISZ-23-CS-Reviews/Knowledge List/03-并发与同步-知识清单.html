一、信号（Signal）

1.1 信号的基本概念 必考

信号（Signal）是一种软件形式的异常控制流（ECF），由内核发送给进程，通知进程发生了某个系统事件。信号是进程间通信（IPC）的一种机制。

常见信号 ID：

  • SIGINT（2）：来自键盘的中断（Ctrl-C）
  • SIGKILL（9）：强制终止进程，不可被捕获或忽略
  • SIGSEGV（11）：段错误（非法内存访问）
  • SIGALRM（14）：定时器到期
  • SIGCHLD（17）：子进程停止或终止
易错：SIGKILL 和 SIGSTOP 是唯二不可被捕获、阻塞或忽略的信号。

1.2 发送信号 重点

进程组（Process Group）：每个进程都属于一个进程组，进程组 ID（PGID）默认等于父进程的 PGID。

发送方式：

  • /bin/kill -signal pid：向指定进程或进程组发送信号；pid 为负数时发送给整个进程组
  • kill(pid, sig) 系统调用：程序内发送信号
  • 键盘快捷键：Ctrl-C --> SIGINT，Ctrl-Z --> SIGTSTP，Ctrl-\ --> SIGQUIT
  • alarm(n)：n 秒后向自身发送 SIGALRM

1.3 接收信号与处理 必考

进程在从内核模式切换回用户模式时检查信号。对信号的三种反应：

  1. 忽略（Ignore）：不做任何处理
  2. 终止（Terminate）：进程被终止（默认行为）
  3. 捕获（Catch）：执行用户自定义的信号处理函数（handler）

pending（待处理）位向量：内核为每个进程维护，收到信号则置 1，处理后清零。

blocked（阻塞）位向量：进程可以阻塞某些信号，使其不被接收。

实际处理的信号集：pnb = pending & ~blocked

易错：标准信号不排队——如果同类型信号多次到来，bit vector 中只记录"有一个待处理"，多余的信号被丢弃。

1.4 阻塞信号 重点

两种阻塞方式：

  • 隐式阻塞：执行某类信号的 handler 时，内核自动阻塞同类信号
  • 显式阻塞：使用 sigprocmask() + SIG_BLOCK / SIG_UNBLOCK / SIG_SETMASK 修改阻塞集合

1.5 安全的信号处理 必考

信号 handler 与主程序并发运行，须遵循以下准则：

  • G1 简单处理：handler 尽量简单，仅设置标志位
  • G2 仅用 async-signal-safe 函数：write() 安全，printf() 不安全（使用静态缓冲区）
  • G3 保存/恢复 errno：handler 开始时保存，结束时恢复
  • G4 访问共享数据时阻塞所有信号：防止竞争条件
  • G5 用 volatile 声明全局变量：防止编译器优化缓存到寄存器；flag 使用 sig_atomic_t 类型
易错：在 handler 中使用 printf/malloc/exit 等非 async-signal-safe 函数会导致死锁或数据损坏。

1.6 信号的可移植处理 常考

用 sigaction() 替代 signal()（后者行为在不同系统上不一致）。可移植的 handler 安装需：正确设置 sa_flags（如 SA_RESTART），并用 sigemptyset() 初始化信号掩码。

二、线程与共享变量

2.1 线程内存模型 必考

线程（Thread）是进程内的轻量级执行单元，共享进程的地址空间。

变量类别：

  • 全局变量：声明在函数外，整个虚拟地址空间只有 1 个实例，所有线程共享
  • 局部自动变量（local-auto）：声明在函数内（无 static），每个线程各有独立副本（存于各自栈帧）
  • 局部静态变量（local-static）：声明在函数内且有 static，整个地址空间只有 1 个实例，所有线程共享
易错：局部自动变量在线程的私有栈上，但若将栈地址传给其他线程，则可被共享（危险！）。

2.2 不正确的共享变量访问 必考

以 badcnt.c 为例：两个线程各执行 cnt++ 1000 次，期望结果 2000，但实际结果不确定。

cnt++ 对应三条汇编指令：

  1. L（Load）：将 cnt 从内存加载到寄存器
  2. U（Update）：寄存器值加 1
  3. S（Store）：将寄存器值写回内存

两线程的 L/U/S 指令交错执行时，可能产生竞争条件（Race Condition），导致更新丢失。

2.3 进度图与临界区 重点

进度图（Progress Graph）：以二维坐标表示两线程的执行轨迹，横轴为线程 1，纵轴为线程 2。

临界区（Critical Section）：访问共享变量的代码段，同一时刻只能有一个线程执行。

不安全区域（Unsafe Region）：进度图中两线程临界区指令集合的笛卡尔积形成的矩形区域；轨迹经过此区域则产生错误结果。

安全轨迹：不进入不安全区域的执行路径，需要同步机制来保证。

三、信号量（Semaphore）

3.1 信号量的定义 必考

信号量（Semaphore）是一个整数变量，配合 P/V 操作（Dijkstra 提出）实现同步。

两种定义风格：

  • CSAPP 风格：信号量值 非负，P 操作在值为 0 时阻塞等待
  • 龙书（教材）风格：信号量值可为负，负值的绝对值 = 等待该信号量的进程/线程数

P（Proberen / wait / sem_wait）：若 S > 0 则 S--，否则阻塞当前线程

V（Verhogen / signal / sem_post）：S++，若有线程阻塞则唤醒一个

易错：P/V 操作是原子的，不可中断。P 减少信号量，V 增加信号量。

3.2 互斥锁（Mutex） 必考

互斥信号量（Mutex）是初值为 1 的二值信号量，用于保护临界区实现互斥。

使用模板：

sem_init(&mutex, 0, 1);   // 初始化为 1
// 线程代码：
P(&mutex);   // 加锁（lock）
// 临界区
V(&mutex);   // 解锁（unlock）

Pthreads API：sem_init()、sem_wait()（P）、sem_post()（V）

3.3 进程制约关系 重点

进程间的制约关系分为两类：

  • 间接制约（互斥）：多个进程竞争同一资源（如打印机），需要互斥访问
  • 直接制约（同步）：多个进程协作完成任务，存在先后顺序关系

同步机制需满足四条准则（补充解释）：

  1. 空闲让进：临界区空闲时，允许一个进程立即进入
  2. 忙则等待：临界区已有进程时，其他进程必须等待
  3. 有限等待：等待进入临界区的进程必须在有限时间内得到满足
  4. 让权等待：无法进入临界区时，应放弃 CPU（不能忙等）

3.4 生产者-消费者问题 必考

经典同步问题，生产者往缓冲区放数据，消费者取数据。

单缓冲区方案（3 个信号量）：

  • empty（初值 1）：缓冲区空槽数
  • full（初值 0）：缓冲区满槽数
  • mutex（初值 1）：互斥锁

生产者：P(empty) --> P(mutex) --> 放数据 --> V(mutex) --> V(full)

消费者：P(full) --> P(mutex) --> 取数据 --> V(mutex) --> V(empty)

N 缓冲区（sbuf 包）方案：

  • slots（初值 N）：空槽数
  • items（初值 0）：已有数据项数
  • mutex（初值 1）：保护缓冲区操作
易错：P(slots)/P(items) 必须在 P(mutex) 之前执行，否则会发生死锁（持锁等待资源）。

3.5 读者-写者问题 必考

多个读者可同时读，写者需独占访问。

第一类（读者优先）：读者不等待，只要有读者在读，新读者可直接进入；写者可能饥饿。

  • readcnt：当前读者数量（共享变量，需 mutex 保护）
  • mutex（初值 1）：保护 readcnt
  • w（初值 1）：写者/首个读者互斥锁
  • 第一个读者 P(w)，最后一个读者 V(w)

第二类（写者优先）：写者等待时不允许新读者进入；读者可能饥饿。需额外信号量 S/S2/wmutex/mutex1/mutex2 等（补充解释）。

易错：读者优先下写者可能永久饥饿；写者优先下读者可能永久饥饿。

3.6 哲学家就餐问题 重点

5 位哲学家围坐，每人需要左右两根筷子才能吃饭。

错误方案：所有人同时拿左筷子 --> 全部等待右筷子 --> 死锁。

三种解决方案：

  1. 限制人数法：用计数信号量 sm=4，限制同时就餐尝试的哲学家不超过 4 人，至少有一人可以吃饭
  2. 同时拿两根：只有两根筷子都可用时才拿，否则不拿任何一根（原子操作）
  3. 奇偶序号法：奇数号哲学家先拿左筷子，偶数号先拿右筷子，打破循环等待

3.7 线程安全 重点

四类非线程安全函数：

  1. 不保护共享变量的函数（如 badcnt）
  2. 在不同调用间保持状态的函数（如 rand()，使用静态变量）
  3. 返回指向静态变量指针的函数（如 ctime()、gethostbyname()）
  4. 调用非线程安全函数的函数

可重入函数（Reentrant Function）：函数内不引用任何共享数据（无全局变量、无静态变量），是线程安全函数的子集，执行效率更高（无需加锁）。

易错：线程安全 ≠ 可重入；可重入 ⊂ 线程安全。可重入函数不依赖任何共享状态。

3.8 竞争条件的消除 重点

竞争条件（Race Condition）：程序的正确性依赖线程的相对执行速度时产生。

消除方法：将栈上的局部变量值通过 malloc 分配到堆上，每次传递独立副本的指针，避免多个线程读同一个栈变量。

// 有竞争（传指针到局部变量）
int i;
pthread_create(&tid, NULL, thread, &i);

// 无竞争（传堆上独立副本）
int *p = malloc(sizeof(int));
*p = i;
pthread_create(&tid, NULL, thread, p);
// thread 内部使用后 free(p)

3.9 信号量与信号的区别 了解

两者名字相近但本质不同：

  • 信号（Signal）：操作系统内核向进程发送的通知机制，属于 IPC（进程间通信），用于处理异步事件
  • 信号量（Semaphore）：用于控制并发访问的同步原语，是计数器 + P/V 操作，用于协调线程/进程对资源的访问

四、同步问题解题方法

4.1 解题四步法 必考

解决同步问题的标准步骤（补充解释）：

  1. 识别线程/进程：确定有哪些并发执行的实体
  2. 判断制约关系：区分互斥（竞争共享资源）和同步（协作依赖顺序）
  3. 定义信号量：为每种制约关系设置合适的信号量及初值
  4. 编写伪代码：在正确位置插入 P/V 操作，确保互斥和同步正确
易错：互斥信号量初值为 1，同步信号量初值通常为 0（表示等待某事件发生）。

4.2 典型练习：博物馆容量控制 常考

博物馆限容 N 人，进入前检查是否有空位，出来后释放名额。

使用计数信号量 capacity（初值 N）：

  • 游客进入：P(capacity)（若已满则等待）
  • 游客离开：V(capacity)（释放名额）

4.3 典型练习：航空订票系统 常考

多个售票员并发操作余票数量 seats，需要互斥保护。

P(mutex)        // 加锁
temp = seats    // 读余票
if temp > 0:
    seats = temp - 1   // 减票
    // 出票
V(mutex)        // 解锁
